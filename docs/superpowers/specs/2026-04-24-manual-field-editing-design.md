# 规格说明：手动字段编辑与审计功能

## 1. 背景与目的
为了防止 AI 自动化处理过程中的识别错误（如学号提取错误、作业归类错误），本功能允许用户在 UI 界面上手动修改作业提交记录的所有核心字段。这不仅能纠正错误，还能为人工审计提供支持。

## 2. 交互设计 (UI/UX)
修改功能集成在现有的“邮件预览侧边栏” (`EmailPreviewDrawer`) 中。

### 2.1 模式切换
- **浏览模式 (Default)**：保持现有展示方式。
- **编辑模式**：点击侧边栏顶部的“编辑”按钮进入。进入后：
    - 文本标签转换为输入框 (`CTkEntry`) 或下拉菜单 (`CTkOptionMenu`)。
    - 顶部显示“保存”和“取消”按钮。

### 2.2 可编辑字段
| 字段 | 控件类型 | 说明 |
| :--- | :--- | :--- |
| 学号 | `CTkEntry` | 修改后触发学生关联逻辑 |
| 姓名 | `CTkEntry` | 修改后触发学生信息更新 |
| 邮箱 | `CTkEntry` | 仅支持手动修正 |
| 作业名称 | `CTkOptionMenu` | 支持选择现有作业或输入新作业名 |
| 状态 | `CTkOptionMenu` | 对应 `SubmissionStatus` 中的所有状态 |

## 3. 技术实现

### 3.1 数据库逻辑 (`database/operations.py`)
新增 `update_submission_full` 方法，处理以下逻辑：
1. **学生关联**：
   - 检查学号是否存在于 `students` 表。
   - 若存在，更新该学生姓名并将提交关联至该学生。
   - 若不存在，创建新学生记录。
2. **作业关联**：
   - 检查作业名是否存在于 `assignments` 表。
   - 若不存在，创建新作业记录。
3. **状态更新**：直接更新 `submissions` 表的 `status` 字段。
4. **事务处理**：确保上述操作在同一个数据库事务中完成。

### 3.2 UI 组件更新 (`EmailPreviewDrawer`)
- 新增 `is_edit_mode` 状态变量。
- 重构 `_update_student_card` 和 `_update_assignment_card`，使其支持两种模式的渲染。
- 实现 `_on_save_clicked`：收集输入数据，调用数据库接口，刷新 UI。

## 4. 验证标准
- 修改学号后，数据库中的 `student_id` 外键应正确更新。
- 修改作业名称为新作业时，系统应自动在 `assignments` 表中增加记录。
- 保存后，主界面表格中的相应行应立即显示更新后的数据。
- 点击“取消”后，界面应恢复原始数据，不做任何持久化更改。
