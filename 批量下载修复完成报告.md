## 问题诊断

### 错误信息
```
Error fetching email 9: command FETCH illegal in state AUTH, only allowed in states SELECTED
```

### 根本原因（已确认）

通过测试脚本发现了两个关键问题：

1. **IMAP状态问题** ✓ 已修复
   - IMAP需要在SELECTED状态下才能执行FETCH命令
   - 修复：在fetch_email等方法中添加自动选择文件夹的逻辑

2. **编码问题** ✓ 已修复
   - 邮件主题包含emoji等特殊字符（如📐）
   - Windows控制台使用GBK编码，无法显示这些字符
   - 导致print语句失败，中断整个流程
   - 修复：添加try-except捕获编码错误

3. **测试数据问题** ✓ 已清理
   - 数据库中有测试UID（test_uid_001），在真实邮箱中不存在
   - 导致获取邮件失败
   - 修复：清理了所有测试数据

## 注意事项

### 关于附件

批量下载**只会下载有附件的邮件**：
- ✓ 有附件 → 下载并保存到本地
- ✗ 无附件 → 标记为"无附件"（这是正常的）

### 关于UID

数据库中的UID必须与邮箱中的邮件对应：
- ✓ 真实邮件UID（1, 2, 3...）→ 可以下载
- ✗ 测试UID（test_uid_001）→ 无法获取

### 关于编码

- 邮件主题可能包含特殊字符（emoji、特殊符号等）
- 已添加错误处理，不会中断流程
- 附件文件名也会被正确处理

## 故障排除

### 问题：所有记录都显示"无附件"

**可能原因：**
1. 这些邮件确实没有附件
2. 附件提取失败（格式问题）

**解决方法：**
1. 检查邮箱中这些邮件是否真的有附件
2. 查看控制台的DEBUG信息
3. 手动下载一封邮件验证

### 问题：某些UID获取失败

**可能原因：**
1. 邮件已被删除
2. UID不正确
3. 邮件在其他文件夹中

**解决方法：**
1. 确认邮件在INBOX中
2. 检查邮件是否被移动或删除
3. 验证数据库中的UID是否正确

### 问题：编码错误中断流程

**状态：** 已修复
- 添加了try-except处理
- 不会因为编码问题中断
